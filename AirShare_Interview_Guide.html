What is AirShare?
Project Overview & Purpose

🎯 One-Line Definition

AirShare is a cross-platform, serverless, peer-to-peer local file transfer application — similar to Apple AirDrop — that lets two devices on the same Wi-Fi network discover each other automatically and transfer files of any size directly without any internet connection, cloud storage, or server infrastructure.

🌍 Real-World Problem It Solves

Sharing files between devices today is unnecessarily complex. You either need:

❌ Cloud uploads (Google Drive, WhatsApp): File goes to a server in America and comes back — wasting bandwidth and time for devices sitting next to each other.
❌ USB cables: Requires physical connectors, drivers, and compatible ports.
❌ AirDrop: Only works between Apple devices.
❌ Bluetooth: Extremely slow — typically under 3 MB/s.

✅ AirShare transfers files directly at your local Wi-Fi speed (potentially 50-200 MB/s) without touching the internet, without any account login, and works across Android and Windows simultaneously.

📱 How It Looks to the User

You open AirShare on your Windows laptop. A dark radar screen appears with a rotating green scan line. Your phone appears as a floating node on the radar within 2-3 seconds. You tap the phone node, a file picker opens, you select a file — and it begins transferring instantly. On your phone, a dialog pops up: "Accept file from Sanjay's Laptop?" You tap Accept, and the file appears in your Downloads folder at full Wi-Fi speed.

How Does It Work?
Step-by-Step Technical Flow

Phase 1 — Device Discovery (UDP Broadcasting)

When AirShare opens, it does two things simultaneously using the UDP (User Datagram Protocol):


App Opens
→
Bind to UDP Port 53530
→
Broadcast JSON packet every 2 seconds to 255.255.255.255
→
All AirShare devices on same Wi-Fi hear it
→
Appear on Radar

The broadcast packet is a tiny JSON message (~200 bytes) containing the device's name, unique ID, IP address, and the TCP port to connect for file transfers.

Phase 2 — File Transfer (TCP Streaming)

Once you tap a peer and select a file, a direct TCP (Transmission Control Protocol) connection is established:


Sender connects to Receiver's IP:53531
→
Sends JSON header (filename, size)
→
Receiver shows Accept/Reject dialog
→
On Accept: file streams in binary chunks
→
Receiver writes to Downloads folder

The file is split into small chunks and streamed continuously. The app displays real-time speed (in MB/s) and a progress bar. On large files, backpressure control is applied — the sender waits for the OS network buffer to drain before sending the next batch, preventing memory overflow.

Phase 3 — State Management (Riverpod)

All application state — which peers are active, which transfers are running, what their speed and progress is — is managed using Riverpod, a reactive state management library. When a new UDP packet arrives from a peer, the peer list updates and the radar UI rebuilds automatically. When transfer progress updates, only the affected transfer card rebuilds — not the entire screen. This ensures 60fps smooth UI at all times.

Technologies Used & Why
Tech Stack Deep Dive
Flutter & Dart
Core Framework
Allows writing a single codebase that runs natively on Android, Windows, iOS, and macOS simultaneously. Dart's async/await model perfectly fits network programming (reading sockets, listening to streams). Flutter's rendering engine (Skia) allows custom canvas drawing for the radar animation.
dart:io (RawDatagramSocket)
UDP Discovery Layer
Dart's built-in networking library exposes raw UDP socket APIs with no third-party dependency. We use RawDatagramSocket to bind to port 53530 and broadcast subnet-wide packets at 2-second intervals. Using UDP (not TCP) is deliberate — UDP is connectionless and ideal for one-to-many broadcast-style announcements.
dart:io (ServerSocket / Socket)
TCP Transfer Engine
TCP guarantees delivery and ordering of all bytes, which is critical for file integrity. ServerSocket accepts incoming connections, reads the file header, and streams chunks directly to disk without loading the entire file into memory — making it capable of transferring files of any size (even 50GB+ videos) without crashing.
flutter_riverpod
State Management
Riverpod provides type-safe, testable, and modular state management. It separates business logic (networking) from UI cleanly. StateNotifierProviders manage peer lists and transfer maps reactively. When state changes, only dependent widgets rebuild — ensuring high performance with dozens of concurrent transfers.
CustomPainter (Canvas API)
Radar Animation
Flutter's CustomPainter gives direct access to the GPU-accelerated Skia canvas. We use it to draw concentric radar rings, the rotating sweep gradient, and pulsing circles using trigonometric functions (sin/cos) — all at 60fps with AnimationController. This demonstrates advanced Flutter rendering knowledge beyond standard widgets.
network_info_plus
Local IP Resolution
Resolves the device's current local IP address (e.g. 192.168.1.5) assigned by the Wi-Fi router. This IP is embedded in each UDP broadcast packet so other devices know exactly which address to connect to when initiating a TCP file transfer. Without this, the app cannot determine its own network identity.
file_picker
File Selection
Provides a native OS file picker dialog on each platform — Windows File Explorer on PC and Android's storage browser on phone. Returns the absolute file path which we use to open a file stream and begin reading/sending binary chunks.
path_provider
Cross-Platform Save Paths
Each operating system stores downloaded files in different locations: Android uses /storage/emulated/0/Download, Windows uses C:\Users\...\Downloads. path_provider resolves the correct platform-specific directory automatically so received files always land in the expected location on every OS.

⚠️ Current Limitations (Honest Assessment for Interviews)

1. No encryption in transit: Data travels as raw bytes. Future version can add TLS socket wrapping for security.

2. Same subnet only: Cannot discover devices across different routers or VLANs. mDNS could extend this.

3. No resume on failure: If a transfer is interrupted midway, it must be restarted from the beginning. A resumable chunked protocol (like BitTorrent) could fix this.

4. No sender-side acknowledgement: The sender has no confirmation that the receiver accepted and saved the file correctly.

Key Technical Concepts
Understand these before your interview
📡 UDP vs TCP — Why We Use Both

UDP (User Datagram Protocol): Fire-and-forget. Sends a packet without establishing a connection or waiting for acknowledgement. Fast, lightweight, but packets can be lost. Perfect for our discovery broadcasts — we send our identity every 2 seconds so even if one packet is dropped, the next one arrives.

TCP (Transmission Control Protocol): Establishes a formal connection, guarantees every byte arrives in order, and retransmits lost packets. Slower to set up but reliable. Perfect for file transfer — every byte of a file MUST arrive correctly or the file is corrupted.

🔊 Subnet Broadcasting — How 255.255.255.255 Works

When we send a UDP packet to 255.255.255.255, the Wi-Fi router delivers it to every device on the local network simultaneously. This is called a broadcast address. Every device running AirShare on port 53530 wakes up, reads the packet, parses the sender's device information, and adds it to their peer list. It is similar to standing in a room and announcing "I'm here!" — everyone hears it at once.

🚿 Backpressure in TCP Streaming

When sending a large file, you cannot dump all the data into the socket at once — the OS network buffer would overflow and crash. Instead, we read the file in small chunks (typically 8-64 KB), write each chunk to the socket, then call socket.flush() to wait until the OS has actually sent that chunk before reading the next one. This is called backpressure — the receiver's buffer controls the rate at which the sender sends. It is the same principle used by HTTP chunked transfer encoding and video streaming.

⚛️ Reactive State Management with Riverpod

Riverpod uses a Provider-Observer pattern. A StateNotifier holds application state (e.g., the list of active peers). Any widget that reads this provider is automatically rebuilt whenever the state changes. When a UDP packet arrives from a new device at 2:00:05 AM, the peer list StateNotifier is updated, and only the radar widget that watches it rebuilds — not the entire app screen. This is fundamentally similar to React's useState + useContext pattern.

🎨 CustomPainter — GPU Canvas Drawing

Flutter's standard UI toolkit (buttons, text, containers) is built on top of a canvas. CustomPainter lets you draw directly to this canvas using paths, arcs, gradients, and lines. The radar animation uses a SweepGradient — a cone-shaped gradient that rotates — driven by an AnimationController that fires 60 times per second. The shouldRepaint() method ensures the canvas only redraws when the angle changes, preventing unnecessary GPU work.

Interview Q&A — Part 1
Common questions and how to answer them confidently
Q Tell me about this AirShare project. What does it do and why did you build it?
AirShare is a cross-platform, serverless file transfer app I built from scratch using Flutter and Dart. The core idea is simple — if two devices are on the same Wi-Fi network, why should a file travel to a server in another country just to come back to a device sitting next to you? AirShare solves this by sending files directly between devices at full local network speed — no internet, no account, no cost.

I built it because I wanted to demonstrate real systems networking skills. Anyone can make a CRUD app. But implementing UDP broadcast discovery and TCP socket streaming from scratch requires understanding how networks actually work at the protocol level — which is exactly the kind of depth product companies look for.
Lead with the problem you solved, not the technology. Interviewers want to see product thinking first.
Q How does the device discovery work? How does your phone "find" the laptop?
Each device runs a UDP socket listener on port 53530. Every 2 seconds, it broadcasts a small JSON packet to the subnet broadcast address 255.255.255.255 — which the router delivers to every device on the network simultaneously.

The packet contains the device's name, unique ID, local IP address, and the TCP port it's listening on for file transfers. When another AirShare device receives this packet, it parses it and adds the sender to its peer list. If a device hasn't broadcast in 8 seconds — meaning it closed the app or lost Wi-Fi — it's automatically pruned from the list.

I chose UDP over TCP for discovery specifically because UDP is connectionless and supports subnet broadcasting. TCP requires a pre-established connection, so you can't use it to announce yourself to unknown devices.
This answer demonstrates understanding of UDP vs TCP tradeoffs. Expect a follow-up: "What if a UDP packet is lost?" Answer: "The next broadcast arrives in 2 seconds anyway, so it self-heals."
Q How do you handle large files? What if the file is 5GB?
I never load the entire file into memory. Instead, I use Dart's file.openRead() which returns a Stream of byte chunks. Each chunk (typically 8-64 KB) is piped directly into the TCP socket, and I call socket.flush() after each chunk to apply backpressure — meaning the sender pauses until the OS confirms that chunk was transmitted before reading the next one.

On the receiver side, I open a file sink and write each incoming chunk directly to disk as it arrives. So at any point in time, only one small chunk is in memory — not the full file. A 5GB file would transfer just as reliably as a 1KB text file. Memory usage stays constant regardless of file size.
This answer shows you understand streaming vs buffering — a key systems concept. Strong follow-up: mention that this is the same principle used by Netflix's video streaming and HTTP chunked transfer encoding.

Interview Q&A — Part 2
Deeper technical and design questions
Q Why did you choose Flutter instead of native Android (Kotlin) or React Native?
The core value proposition of AirShare is that it works across all platforms from one codebase. If I built it in native Android Kotlin, I'd have to rewrite the entire networking layer, UI, and state management separately for Windows, iOS, and macOS. Flutter gives true native compilation on all platforms from a single Dart codebase.

Additionally, Dart's async/await and Stream APIs are first-class citizens — which maps perfectly to TCP socket programming where you're constantly listening to streams of incoming bytes. And Flutter's CustomPainter gave me direct canvas access to build the radar animation without needing any third-party library.

React Native was ruled out because it depends on a JavaScript bridge to native code — introducing latency in the hot path of our file streaming loop. Flutter compiles to native ARM code directly.
Demonstrating that you evaluated multiple options and chose based on technical merit shows seniority. Always explain trade-offs, not just your choice.
Q What state management approach did you use and why?
I used Riverpod with StateNotifier. The peer list and active transfers are each managed by separate StateNotifierProviders. When a UDP packet arrives from a new device, the PeerListNotifier updates its state and only widgets subscribed to that provider rebuild — not the entire widget tree.

I chose Riverpod over alternatives like Provider or BLoC because it's compile-time safe — you can't accidentally access a provider that hasn't been initialised. It also separates business logic from UI cleanly, which makes unit testing the networking layer possible without spinning up a UI at all.
Interviewers at product companies love hearing about testability. Mention that you wrote unit tests for the Peer model serialisation to verify that UDP packet parsing works correctly.
Q What would you add or improve if you had more time?
Several improvements would make this production-ready:

1. End-to-End Encryption: Wrap the TCP socket with TLS using a self-signed certificate exchanged over the discovery broadcast. This would prevent any device on the network from sniffing the file contents.

2. Resumable Transfers: Store the byte offset of interrupted transfers. On reconnection, the sender can start streaming from where it stopped — similar to how download managers work.

3. Multi-file & folder transfer: Currently one file at a time. ZIP compression on the fly would enable folder transfers.

4. mDNS Discovery: Replace UDP broadcast with mDNS (Multicast DNS) for more reliable peer discovery across complex network topologies including some enterprise networks that block broadcasts.
This question tests intellectual honesty and product maturity. Saying "nothing, it's perfect" is a red flag. Showing a clear technical roadmap demonstrates engineering leadership mindset.
Q How did you handle errors and edge cases?
Several edge cases were handled deliberately:

File name collisions: If a file with the same name already exists in the Downloads folder, the app automatically appends a counter suffix (e.g. photo_1.jpg, photo_2.jpg) instead of overwriting silently.

Peer timeout: If a device closes the app or loses Wi-Fi, it stops broadcasting. Any peer not heard from in 8 seconds is pruned from the radar automatically — the UI stays accurate.

Connection refused: If the receiver rejects the file, the sender gets a REJECT response over the TCP handshake and transitions the transfer status to Rejected — not an error crash.

Socket errors: All socket operations are wrapped in try-catch blocks. Network errors surface as user-visible error messages on the transfer card, not as application crashes.
Error handling depth signals production engineering maturity. Senior engineers always think about what happens when things go wrong, not just the happy path.
